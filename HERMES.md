# HERMES.md - Hermes 办公协同 Agent 核心原则与规范

> 本文档定义了 Hermes 办公协同 Agent 的开发、运维和自我进化的核心原则，
> 确保系统稳定、安全、可控，符合企业级合规要求。

---

## 一、四大核心原则

### 1. Think Before Coding：编码前先想清，不脑补、不隐藏困惑

**核心思想**：在执行任何操作前，必须明确目标、假设和风险

**具体实践**：
- **技能生成必须加假设澄清**：草稿生成前自动分析用户需求、歧义点、资源需求、权限要求、异常情况和合规风险
- **意图识别必须显式输出**：对模糊需求主动追问确认，例如："你想整理哪类文件？是否包含子目录？"
- **插件接入前先声明边界**：明确支持的事件类型、权限范围、异常处理策略

**在项目中的实现**：
- `learning_cycle.py` 中的 `_clarify_assumptions()` 方法自动生成假设清单
- 假设清单包含：核心需求、歧义点、资源列表、权限需求、潜在异常、合规风险、置信度评分

---

### 2. Simplicity First：最小代码解决问题，拒绝过度设计

**核心思想**：够用就好，不为"未来可能有用"做准备

**具体实践**：
- **技能功能最小化**：用户只要求"每周一生成周报"，就不做"多模板、多部门、定时推送"的大系统
- **避免不必要的抽象**：一次性脚本不套插件基类，能用50行搞定就不写200行
- **异常处理聚焦**：只处理办公场景中真可能发生的情况（网络超时、权限不足、文件不存在）

**在项目中的实现**：
- `skill_manager.py` 中的 `check_complexity()` 方法检查技能复杂度阈值
- 复杂度超过阈值自动触发人工审核

**复杂度阈值表**：

| 指标 | 阈值 | 超过处理 |
|------|------|----------|
| 步骤数 | ≤ 10 | 触发人工审核 |
| 条件分支数 | ≤ 3 | 触发人工审核 |
| 嵌套深度 | ≤ 2 | 触发人工审核 |
| 工具调用数 | ≤ 5 | 触发人工审核 |

---

### 3. Surgical Changes：精准修改，只碰必须改的，不乱优化

**核心思想**：每一行变更都必须有明确的理由，且能追溯到用户请求

**具体实践**：
- **严格"最小diff"**：只改用户要求的那一步，不顺手优化注释、格式、变量名
- **不删无关代码**：对无用代码只标注为 deprecated，不直接删除
- **自己清理自己的垃圾**：自己引入的无用导入/变量必须自己清理

**在项目中的实现**：
- `skill_manager.py` 中的 `validate_skill_changes()` 方法验证变更是否符合最小diff原则
- 变更超过3个自动触发警告，拒绝无关变更
- 所有变更记录到审计日志，包含：请求ID、操作人、权限、时间

---

### 4. Goal-Driven Execution：目标可验证，用测试闭环保证质量

**核心思想**：所有需求必须转化为可验证的目标，通过测试闭环保证质量

**具体实践**：
- **模糊需求转可验证目标**：
  - "加权限" → 写用例：普通用户不能删技能、admin可以 → 用例通过才算完成
  - "修bug" → 先写复现用例 → 修复 → 用例通过
- **多步骤任务加校验点**：每个步骤都有明确的验证机制
- **自我进化可验收**：新技能上线前必须通过固定测试集

**在项目中的实现**：
- `tests/test_agent_self_verification.py` 包含完整的自验证用例库
- 学习循环三闸门确保技能质量：假设澄清 → 复杂度检查 → 测试验证

## 二、技能生成规范

### 1. 学习循环三闸门机制

```
用户反馈 → [闸门1:假设澄清] → [闸门2:复杂度检查] → [闸门3:测试验证] → 技能入库
```

**闸门1：假设澄清**
- 自动分析用户意图、上下文、潜在歧义
- 生成 `AssumptionChecklist` 供管理员审核
- 置信度低于70%时拒绝继续

**闸门2：复杂度检查**
- 检查步骤数、分支数、嵌套深度、工具调用数
- 超过任一阈值自动进入"待审核"状态

**闸门3：测试验证**
- 自动运行技能测试用例
- 测试通过率100%才能正式入库

### 2. 技能质量标准

| 指标 | 要求 | 说明 |
|------|------|------|
| 测试通过率 | 100% | 自动测试必须全部通过 |
| 步骤数 | ≤ 10 | 超过需人工审核 |
| 执行成功率 | ≥ 95% | 上线后持续监控 |
| 用户满意度 | ≥ 4.5/5 | 基于用户反馈计算 |

### 3. 技能变更规则

- **变更必须关联用户请求**：每一次技能修改都必须有对应的用户反馈或管理员操作
- **变更内容限制**：同一版本变更不超过3处（名称、描述、步骤、触发模式等）
- **版本管理**：每次变更自动递增版本号，保留完整历史
- **回滚支持**：支持回滚到任意历史版本

## 三、插件开发规范

### 1. 白名单机制

**插件类型与白名单**：

| 插件类型 | 允许的插件 | 对应模块 |
|----------|------------|----------|
| IM适配器 | feishu, dingtalk, wecom, wechat, slack, discord | `src/gateway/` |
| 模型路由 | openai, anthropic, ollama, zhipu, moonshot | `src/plugins/model_routers.py` |
| 记忆存储 | chroma, milvus, faiss | `src/plugins/memory_stores.py` |
| 技能管理 | default | `src/skills/skill_manager.py` |

**危险工具列表**（需管理员授权）：
- `file_delete` - 文件删除
- `system_command` - 系统命令执行
- `network_scan` - 网络扫描
- `data_export` - 数据导出

### 2. 接口规范

所有插件必须继承对应的抽象基类：

| 插件类型 | 基类 | 必须实现的方法 |
|----------|------|----------------|
| IM适配器 | `IMAdapterBase` | `start()`, `stop()`, `send_message()`, `get_adapter_type()` |
| 模型路由 | `ModelRouterBase` | `select_model()`, `call_model()`, `get_model_type()` |
| 记忆存储 | `MemoryBase` | `add_memory()`, `search_memory()`, `delete_memory()`, `get_memory_type()` |
| 技能管理 | `SkillManagerBase` | `add_skill()`, `get_skill()`, `execute_skill()`, `delete_skill()` |

### 3. 安全要求

- **配置验证**：插件配置加载前必须经过安全验证，检查敏感信息
- **权限声明**：插件必须明确声明所需权限
- **异常处理**：统一异常处理，不泄露敏感信息

## 四、权限与审计规范

### 1. 角色体系

| 角色 | 权限范围 | 适用场景 |
|------|----------|----------|
| `admin` | 全权限（技能、工具、记忆、API、配置） | 系统管理员 |
| `developer` | 技能全权限、工具全权限、记忆读写、API访问、配置查看 | 开发人员 |
| `user` | 技能读写执行、工具执行、记忆读取、API访问 | 普通员工 |
| `guest` | 仅技能读取权限 | 外部访客 |

### 2. 权限类型

| 资源类型 | 权限 | 说明 |
|----------|------|------|
| 技能 | read, execute, edit, delete, grant | 技能的访问和管理 |
| 工具 | execute, configure | 工具执行和配置 |
| 记忆 | read, write, delete, search | 记忆存储访问 |
| API | access | API接口访问 |
| 配置 | view, modify | 系统配置 |

### 3. 权限范围

- `user`：针对特定用户
- `department`：针对部门所有用户
- `all`：全局权限

### 4. 审计日志规范

**记录的操作类型**：

| 类别 | 操作类型 | 记录内容 |
|------|----------|----------|
| 用户管理 | login, logout, user_create, role_change | 用户ID、时间、IP地址、结果 |
| 技能管理 | skill_create, skill_edit, skill_delete, skill_execute | 技能ID、操作人、变更内容 |
| 工具操作 | tool_execute, tool_configure | 工具ID、参数摘要、权限检查结果 |
| 权限管理 | permission_grant, permission_revoke | 资源类型、用户ID、权限列表 |
| 系统配置 | config_modify | 配置项、原值、新值 |

**防篡改机制**：
- SHA-256哈希链：每条日志包含前一条的校验和
- 支持完整日志链验证
- 日志文件只读存储

## 五、自我进化规范

### 1. 学习流程

```
用户反馈 → 意图提取 → 差异分析 → 假设澄清 → 技能草稿 → 复杂度检查 → 自动验证 → 人工审核 → 技能入库
```

### 2. 进化控制策略

- **自动生成的技能默认为"待审核"状态**：必须经过管理员确认才能正式生效
- **定期技能质量评估**：每周自动评估技能执行成功率，低于95%的技能标记为"待优化"
- **低质量技能清理**：连续4周执行成功率低于90%的技能自动归档

### 3. 技能生命周期

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| draft | 草稿状态 | 刚生成的技能草稿 |
| pending_review | 待审核 | 自动验证通过后 |
| approved | 已批准 | 管理员审核通过 |
| rejected | 已拒绝 | 管理员审核拒绝 |
| archived | 已归档 | 长期未使用或低质量 |

## 六、代码审查标准

### 1. 必查项

| 检查项 | 标准 |
|--------|------|
| HERMES.md 原则 | 是否符合四大核心原则 |
| 测试覆盖 | 新增代码必须有对应的测试用例 |
| 安全漏洞 | 无SQL注入、无敏感信息泄露 |
| 性能问题 | 无明显性能瓶颈 |
| 变更范围 | 变更是否最小化，无无关修改 |

### 2. 拒绝合并的情况

- 未通过测试用例
- 变更超出需求范围（多余的优化）
- 没有对应的审计日志记录
- 破坏现有功能（回归测试失败）
- 技能复杂度超过阈值且未申请人工审核

## 七、运维与监控规范

### 1. 日志标准

**日志文件结构**：

| 文件 | 内容 | 轮转策略 |
|------|------|----------|
| `logs/api.log` | API请求日志，包含请求ID、用户ID | 每日轮转 |
| `logs/model.log` | 模型调用日志，包含耗时、令牌数 | 每日轮转 |
| `logs/im.log` | IM消息日志，包含消息路由、推送 | 每日轮转 |
| `logs/engine.log` | 引擎日志，包含技能执行、学习循环 | 每日轮转 |
| `logs/audit.log` | 审计日志（不可篡改） | 每月轮转，永久保存 |

**日志格式**：
```
[时间戳] [请求ID] [用户ID] [模块] [级别] [消息] [堆栈信息(可选)]
```

### 2. 监控指标

| 指标 | 目标值 | 告警阈值 |
|------|--------|----------|
| 技能执行成功率 | ≥ 95% | < 90% |
| API响应时间 | < 3秒 | > 10秒 |
| 错误率 | < 1% | > 5% |
| 用户满意度 | ≥ 4.5/5 | < 4.0/5 |
| 模型调用成功率 | ≥ 99% | < 95% |

### 3. 告警规则

- **技能执行失败率 > 5%**：通知运维人员
- **响应时间 > 10秒**：立即告警
- **权限异常访问**：实时通知安全人员
- **日志完整性验证失败**：立即告警并停止写入

## 九、合规要求

### 1. 数据隐私

- **数据加密**：用户数据（包括记忆、对话记录）加密存储
- **敏感信息脱敏**：日志中的敏感信息（如API Key、密码）自动脱敏
- **访问审计**：所有数据访问操作记录到审计日志

### 2. 安全审计

- **定期安全扫描**：每周自动扫描代码漏洞
- **权限定期审查**：每月审查用户权限，清理过期权限
- **日志完整性验证**：每日验证审计日志哈希链

### 3. 备份策略

- **数据库备份**：每日增量备份，每周全量备份
- **向量库备份**：每日备份
- **审计日志备份**：异地备份，保留期限根据合规要求

## 附录：术语定义

| 术语 | 定义 |
|------|------|
| 最小diff | 只修改实现需求必须的代码，不做无关变更 |
| 可验证目标 | 可以通过测试用例验证是否完成的目标 |
| 假设澄清 | 在执行前验证理解是否正确的过程，生成AssumptionChecklist |
| 学习循环三闸门 | 假设澄清→复杂度检查→测试验证的三级验证机制 |
| 技能质量评分 | 根据测试通过率、执行成功率、用户满意度计算的分数 |
| 危险工具 | 需要管理员授权才能执行的工具（file_delete、system_command等） |
